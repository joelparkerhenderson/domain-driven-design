# Anti-Corruption Layer in Cardiology

## Overview

An anti-corruption layer (ACL) is a translation boundary that protects a bounded context from the concepts, terminology, and data structures of external systems. In the cardiology domain, anti-corruption layers are critical because cardiology systems must integrate with numerous external platforms -- electronic health records (EHRs), laboratory information systems (LIS), pharmacy systems, billing platforms, and medical device interfaces -- each with its own data model that does not align with cardiology's ubiquitous language.

## Purpose

Without anti-corruption layers, external system models would leak into cardiology bounded contexts, corrupting the domain model with foreign concepts. For example, an EHR system might represent a cardiac biomarker result as a generic "lab observation" with a coded value, while the Clinical Assessment Context needs a strongly typed cardiac biomarker value object with clinical interpretation logic. The ACL performs this translation at the boundary.

## Key Anti-Corruption Layers

### EHR Integration ACL

The EHR integration ACL sits between all cardiology bounded contexts and the organization's electronic health record system. It translates between the EHR's generic clinical data model and the cardiology-specific models.

**Incoming translations**: Patient demographic data is mapped to cardiology patient entities. Clinical notes are parsed for relevant cardiac findings. Medication lists are filtered and translated into GDMT-relevant therapy value objects. Problem lists are mapped to cardiology-specific diagnosis entities.

**Outgoing translations**: Cardiology assessment results, imaging reports, and procedural notes are translated from the cardiology domain model into the EHR's document and observation format.

### Laboratory Information System ACL

The LIS ACL translates laboratory results into cardiac biomarker value objects used by the Clinical Assessment Context and Heart Failure Management Context.

**Incoming translations**: Generic lab result messages (HL7 ORU) are parsed to extract troponin values, BNP/NT-proBNP levels, lipid panels, hemoglobin A1c, and renal function markers. Each result is translated into a typed value object with units, reference ranges, and clinical interpretation (normal, elevated, critically elevated).

**Key invariants**: The ACL validates that result units match expected formats, converts between units when necessary (e.g., pg/mL to ng/L for BNP), and flags results that fall outside physiologically plausible ranges.

### Pharmacy System ACL

The pharmacy ACL protects the Heart Failure Management Context from the pharmacy system's medication data model. Pharmacy systems model medications generically (drug name, dose, route, frequency), while the heart failure context needs to understand medications in terms of GDMT classification.

**Incoming translations**: Medication records are classified into GDMT pillars (beta-blocker, ACEi/ARB/ARNI, MRA, SGLT2i). Current doses are compared against target doses. Medication history is translated into titration timeline entities.

**Outgoing translations**: GDMT optimization recommendations are translated into pharmacy-compatible prescription requests.

### Medical Device Interface ACL

The device interface ACL translates between cardiac device manufacturer proprietary data formats and the Electrophysiology Context's domain model.

**Incoming translations**: Pacemaker and ICD interrogation data from various manufacturers (Medtronic, Abbott, Boston Scientific, Biotronik) are normalized into a unified device interrogation value object containing battery status, lead impedances, sensing thresholds, therapy delivery history, and arrhythmia episode logs.

**Outgoing translations**: Device programming recommendations generated by the domain model are translated into manufacturer-specific programming commands.

### Billing and Coding ACL

The billing ACL translates between the cardiology domain's clinical concepts and the billing system's CPT and ICD-10 coding requirements.

**Incoming translations**: Billing system queries for procedure codes are mapped to cardiology procedure entities. Prior authorization requirements are translated into clinical documentation needs.

**Outgoing translations**: Completed procedures, diagnoses, and assessments are translated into appropriate CPT procedure codes and ICD-10 diagnosis codes.

## Design Principles

Anti-corruption layers in the cardiology domain follow these principles:

1. **Isolation**: The ACL is the only point of contact between the external system and the bounded context. No external data model concepts leak past the ACL.
2. **Translation fidelity**: Translations preserve clinical meaning. A troponin value of 0.04 ng/mL must not become 4.0 due to unit conversion errors.
3. **Fail-safe behavior**: When translation fails (unrecognized codes, missing data), the ACL raises domain exceptions rather than passing corrupted data.
4. **Versioning**: ACLs must handle external system version changes without requiring changes to the cardiology domain model.
5. **Auditability**: All translations are logged for clinical audit and regulatory compliance purposes.
