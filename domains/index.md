# Domain Driven Design - Domains

Domain-Driven Design (DDD) creates modular scalable systems by modeling using shared "ubiquitous language" between domain experts, developers, and stakeholders, and by grouping systems into bounded contexts.

## Common Structure

Each domain follows a consistent structure:

- `CLAUDE.md` - Domain-specific instructions and conventions
- `plan.md` - Domain project plan with phases
- `tasks.md` - Task tracking checklist
- `ubiquitous-language.md` - Ubiquitous language glossary
- `index.md` - Main documentation entry point (with `README.md` symlink)
- `topics/` - Detailed topic documentation, each in its own directory

Each topic directory contains:

- `index.md` - Comprehensive documentation with overview, key concepts, examples, relationships, and references
- `README.md` - Symlink to `index.md`

## Project Files

- [plan.md](plan.md) - Master project plan with domain work order and task tracking
- [CLAUDE.md](CLAUDE.md) - Project instructions and conventions
